## 1. 总体开发策略

这个项目不应按“页面”一次性开发，而应按“可验收能力”分阶段推进。每个阶段都必须满足：

- 需求范围明确。
- 数据结构稳定。
- 有自动化测试。
- 有人工验收清单。
- 有性能和安全门禁。
- 阶段完成后可以独立演示。

推荐采用“三层推进”：

1. 横向平台能力：用户、项目、存储、权限、模型、任务、计费。
2. 核心生产链路：剧本、主体、分镜、图像、视频、剪辑、审阅。
3. 高级生产能力：AI 助手、3D 导演台、批量生产、协作、后台。

## 3. 统一研发流程

每个阶段遵循同一套流程：

1. 需求确认
   - 明确本阶段包含/不包含的功能。
   - 明确用户故事和验收场景。
   - 明确依赖的前置模块。

2. 原站依据核验
   - 当 PRD、设计文档、接口描述或验收标准对某个功能的业务逻辑、交互流程、节点选项、模型参数、状态流转、错误处理描述不清晰时，必须先使用 Chrome 插件和 CDP 辅助检查原网站在相同功能上的真实实现和可见行为。
   - 核验范围包括：页面 DOM、可见交互、弹窗、下拉选项、节点配置、网络请求形态、任务状态变化、错误提示、按钮启用条件、默认值和保存结果。
   - 由于原站属于用户所在公司项目，且用户已明确授权，在公司授权范围内，核验结果可以作为直接实现依据，也可以复用对应代码、素材、品牌、图标、样式文件、内置提示词、接口形态和实现细节。
   - 复用时必须记录来源、文件/模块、用途、权限边界和改造点。
   - 涉及密钥、Token、用户数据、支付数据、日志和内部账号信息时不得复制到新项目代码库。
   - 核验结论必须记录到对应任务或阶段验收报告中，格式为：`原站观察 -> 我方实现决策 -> 未确认风险`。
   - 如果原站无法访问、账号权限不足或功能受限，必须明确记录阻塞原因，并给出可验证的产品假设，不能静默猜测。

3. 技术设计
   - 数据结构。
   - API。
   - 状态管理。
   - 任务流。
   - 权限边界。
   - 异常处理。

4. 测试设计
   - 单元测试清单。
   - 集成测试清单。
   - E2E 测试清单。
   - 手动验收清单。
   - 性能基线。

5. 实现
   - 先实现数据模型和 API。
   - 再实现前端状态和 UI。
   - 最后实现异步任务和边界场景。

6. 自测
   - 本地单测。
   - API 集成测试。
   - Playwright 核心路径。
   - 视觉检查。

7. 阶段验收
   - 根据本阶段验收标准逐项验证。
   - 记录已知问题。
   - 阻断问题必须修复。

8. 合并与发布
   - Code Review。
   - 安全检查。
   - CI 通过。
   - 灰度发布。
   - 线上监控。

1. 使用 Chrome 插件观察 UI。
2. 使用 CDP 检查 DOM、Network、Console、Local Storage、接口 payload。
3. 记录截图、接口、状态变化。
4. 生成新时间戳证据文件。
5. 更新 PRD/WBS 对应章节后再开发。

##### 1.2 阶段 Definition of Done 补强

每个阶段除原 WBS DoD 外，新增：

- 所有新增/变更 nodeId 已同步 PRD 6.3 和 WBS 8.3。
