# AI 短剧协作平台完整研发 WBS、测试与验收流程

版本：v0.1
日期：2026-07-01
适用范围：在用户所在公司授权范围内复刻并实现一个 AI 短剧协作平台，包含无限画布、节点系统、AI 生成、主体库、风格一致性、3D 导演台、剪辑预览、成片审阅、协作、计费和后台；原站代码、素材、样式、提示词、接口形态和实现细节可作为直接实现依据，密钥、Token、用户数据、支付数据、日志和内部账号信息不得复制到新项目代码库。

## 1. 总体开发策略

这个项目不应按“页面”一次性开发，而应按“可验收能力”分阶段推进。每个阶段都必须满足：

- 需求范围明确。
- 数据结构稳定。
- 有自动化测试。
- 有人工验收清单。
- 有性能和安全门禁。
- 阶段完成后可以独立演示。

推荐采用“三层推进”：

1. 横向平台能力：用户、项目、存储、权限、模型、任务、计费。
2. 核心生产链路：剧本、主体、分镜、图像、视频、剪辑、审阅。
3. 高级生产能力：AI 助手、3D 导演台、批量生产、协作、后台。

## 3. 统一研发流程

每个阶段遵循同一套流程：

1. 需求确认
   - 明确本阶段包含/不包含的功能。
   - 明确用户故事和验收场景。
   - 明确依赖的前置模块。

2. 原站依据核验
   - 当 PRD、设计文档、接口描述或验收标准对某个功能的业务逻辑、交互流程、节点选项、模型参数、状态流转、错误处理描述不清晰时，必须先使用 Chrome 插件和 CDP 辅助检查原网站在相同功能上的真实实现和可见行为。
   - 核验范围包括：页面 DOM、可见交互、弹窗、下拉选项、节点配置、网络请求形态、任务状态变化、错误提示、按钮启用条件、默认值和保存结果。
   - 由于原站属于用户所在公司项目，且用户已明确授权，在公司授权范围内，核验结果可以作为直接实现依据，也可以复用对应代码、素材、品牌、图标、样式文件、内置提示词、接口形态和实现细节。
   - 复用时必须记录来源、文件/模块、用途、权限边界和改造点。
   - 涉及密钥、Token、用户数据、支付数据、日志和内部账号信息时不得复制到新项目代码库。
   - 核验结论必须记录到对应任务或阶段验收报告中，格式为：`原站观察 -> 我方实现决策 -> 未确认风险`。
   - 如果原站无法访问、账号权限不足或功能受限，必须明确记录阻塞原因，并给出可验证的产品假设，不能静默猜测。

3. 技术设计
   - 数据结构。
   - API。
   - 状态管理。
   - 任务流。
   - 权限边界。
   - 异常处理。

4. 测试设计
   - 单元测试清单。
   - 集成测试清单。
   - E2E 测试清单。
   - 手动验收清单。
   - 性能基线。

5. 实现
   - 先实现数据模型和 API。
   - 再实现前端状态和 UI。
   - 最后实现异步任务和边界场景。

6. 自测
   - 本地单测。
   - API 集成测试。
   - Playwright 核心路径。
   - 视觉检查。

7. 阶段验收
   - 根据本阶段验收标准逐项验证。
   - 记录已知问题。
   - 阻断问题必须修复。

8. 合并与发布
   - Code Review。
   - 安全检查。
   - CI 通过。
   - 灰度发布。
   - 线上监控。

## 4. 全局 Definition of Done

任何功能标记为完成，必须满足：

- 需求在 PRD/WBS 中有对应条目。
- 前端 UI 可操作，不只是静态页面。
- 后端接口有权限校验。
- 数据能持久化，刷新不丢失。
- 有加载、空态、失败态、成功态。
- 有单元测试或集成测试。
- 核心路径有 E2E 覆盖。
- 错误信息对用户友好。
- 日志可定位问题。
- PRD/设计不清晰的功能，已经完成原站依据核验，或明确记录无法核验原因与产品假设。
- 不硬编码密钥。
- 不把模型供应商 API Key 暴露给前端。
- 若复用原站代码、素材、品牌、样式、提示词或实现细节，已经记录来源、权限边界和改造点。
- 不复制密钥、Token、用户数据、支付数据、日志或内部账号信息。

## 5. Phase 0：工程与基础设施

### 5.1 开发任务

- 建立 monorepo。
- 建立前端应用：React、Vite、TypeScript、Tailwind。
- 建立后端 API：NestJS/Fastify 或等价框架。
- 建立 Worker 服务：Node/Python 均可，建议先 Node 统一技术栈。
- 建立数据库：PostgreSQL。
- 建立缓存/队列：Redis、BullMQ。
- 建立对象存储：S3/R2/OSS 抽象。
- 建立本地开发 Docker Compose。
- 建立 ESLint、Prettier、TypeScript strict。
- 建立测试框架：Vitest/Jest、Playwright。
- 建立 CI：lint、typecheck、test、build。
- 建立环境变量规范。
- 建立日志规范。
- 建立 API 响应 envelope。

### 5.2 测试内容

- CI 能在干净环境安装依赖并构建。
- 前端能启动并访问。
- 后端 health check 正常。
- 数据库迁移能执行。
- Redis 连接正常。
- 对象存储 mock 上传/下载正常。

### 5.3 验收标准

- `pnpm dev` 或等价命令能启动前后端。
- `pnpm test` 通过。
- `pnpm build` 通过。
- 新开发者按 README 能在 30 分钟内启动项目。
- `.env.example` 完整，不包含真实密钥。

### 5.4 阶段退出条件

- 工程模板稳定。
- CI 绿色。
- 基础健康检查页面可访问。

## 20. 核心 E2E 验收路径

### 20.1 剧本到视频路径

步骤：

1. 登录。
2. 创建项目。
3. 进入创作画布。
4. 粘贴剧本。
5. 执行剧本整理。
6. 执行主体提取。
7. 保存角色到主体库。
8. 执行分镜生成。
9. 创建图像节点。
10. 选择全局风格。
11. 生成图像。
12. 创建视频节点。
13. 连接图像到视频首帧。
14. 生成视频。

验收：

- 每一步 UI 状态正确。
- 生成结果写入节点。
- 资源库出现图片和视频。
- 刷新后数据不丢失。

### 20.2 3D 到图像路径

步骤：

1. 创建 3D 导演台节点。
2. 进入 3D 导演台。
3. 添加角色模型。
4. 调整相机。
5. 输出截图。
6. 返回画布。
7. 用截图创建图像节点或作为参考图。

验收：

- 3D 页面可进入和退出。
- 模型显示正常。
- 输出截图可用。
- 画布状态不丢失。

### 20.3 视频到成片路径

步骤：

1. 进入剪辑预览。
2. 从素材库拖入视频。
3. 裁剪视频。
4. 添加旁白配音。
5. 播放预览。
6. 导出成片。
7. 进入成片审阅。
8. 标记通过。

验收：

- 时间线保存恢复。
- 导出任务完成。
- 成片审阅列表出现成片。
- 状态流转正确。

### 20.4 团队协作路径

步骤：

1. Owner 创建项目。
2. 邀请 Editor。
3. Editor 进入项目编辑画布。
4. Owner 锁定画布。
5. Editor 变为只读。
6. Owner 解锁。
7. Editor 可继续编辑。

验收：

- 权限准确。
- 锁状态实时同步。
- 无覆盖冲突。

## 24. 最终交付标准

最终产品应满足：

- 一条完整链路：剧本 -> 主体 -> 分镜 -> 图像 -> 视频 -> 剪辑 -> 成片审阅。
- 一个可扩展画布：所有节点可创建、配置、连线、执行、保存。
- 一个可运维模型网关：模型可配置、任务可追踪、成本可控制。
- 一个一致性系统：主体库、全局风格、资源引用可用。
- 一个高级创作入口：3D 导演台可用于镜头构图。
- 一个生产闭环：剪辑、导出、审阅可用。
- 一个商业化底座：团队、权限、计费、后台可用。
- 一套质量体系：自动化测试、E2E、性能、安全、监控完整。
