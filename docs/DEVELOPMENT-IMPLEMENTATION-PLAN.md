# ZeroClock 开发实施计划

## 1. 目标与依据

ZeroClock 当前目标是实现一个 AniShort 类 AI 短剧协作平台：以无限画布为核心，将剧本、主体库、分镜、图像生成、视频生成、3D 导演台、剪辑预览、成片审阅、团队协作和计费治理串成可生产、可验收、可扩展的工作流。

本计划继承 Codex 线程 `019f16f7-680f-7100-976d-56b04d62050e` 的结论。开发主依据为：

- `docs/product/prd/AniShort-PRD-REORGANIZED-IMPLEMENTATION-20260701-171019.md`
- `docs/product/wbs/AniShort-WBS-REORGANIZED-IMPLEMENTATION-20260701-171019.md`

执行中必须同时维护：

- PRD 章节
- WBS Phase
- `nodeId`
- 自动化测试
- 人工验收项
- 原站 Chrome/CDP 复核记录

## 2. 实施原则

1. 先做可闭环能力，再做高级增强。
2. 先稳定节点数据结构、任务状态机和模型网关，再堆具体 AI 能力。
3. 每个 Phase 都要有开发任务、测试、验收标准和退出条件。
4. 涉及不清晰业务逻辑时，必须用 Chrome/CDP 复核原站实现。
5. 新增或修改节点时，必须同步更新 PRD 节点规格矩阵和 WBS 节点级验收矩阵。
6. 安全边界不变：不得迁入密钥、Token、用户数据、支付数据、日志和内部账号信息。

## 3. 推荐技术栈

前端：

- React 19 + TypeScript
- Vite
- Tailwind CSS
- Zustand
- `@xyflow/react` / React Flow
- Radix UI 或 Ant Design
- Three.js
- Monaco Editor 或 TipTap
- HLS.js / Video.js
- WaveSurfer.js
- Vitest + Testing Library + Playwright

后端：

- TypeScript Fastify 或 NestJS
- PostgreSQL
- Prisma 或 Drizzle
- Redis
- BullMQ
- S3/R2/OSS 兼容对象存储
- WebSocket 或 SSE
- FFmpeg Worker
- Python FastAPI Worker 可选，用于媒体处理、OCR 和模型代理

AI 与任务：

- 统一 Model Gateway
- Provider Adapter：text/image/video/audio/media
- JSON Schema 输出校验
- 任务状态机：created -> queued -> running -> succeeded / failed / canceled
- 积分预检查、冻结、扣除、失败返还
- 任务日志和可追踪成本

## 4. 里程碑路线

### Milestone A：可编辑画布

范围：Phase 0、1、2、3。

目标：

- 用户可以创建项目和画布。
- 所有节点类型可以作为空壳创建。
- 节点可以拖拽、连线、配置、保存、恢复。
- 节点 `nodeId` 与 PRD/WBS 对齐。

退出标准：

- 画布保存恢复 E2E 通过。
- 节点库 31 个 `nodeId` 均可创建。
- 节点默认数据结构有单元测试。
- Playwright 覆盖项目创建、画布创建、节点创建、保存恢复。

### Milestone B：剧本到图片

范围：Phase 4、5、6 的核心能力。

目标：

- 剧本输入、剧本分段、主体提取、主体生成、分镜生成可用。
- 图像生成任务可通过 mock 或真实 provider 跑通。
- 主体库、全局风格、资源引用形成第一版一致性系统。

退出标准：

- 从剧本到图片的 E2E 路径通过。
- 主体库去重、别名、类型校验有单元测试。
- Prompt Builder 输出符合 schema。
- 图像任务失败、重试、取消和积分返还逻辑通过集成测试。

### Milestone C：剧本到视频

范围：视频生成、批量基础能力、资源库闭环。

目标：

- 图片、文本、分镜和主体引用可以生成视频。
- 视频任务进入资源库和节点历史。
- 批量生成具备暂停、停止、重试和成本预估。

退出标准：

- 图像到视频 E2E 通过。
- 视频任务轮询或 SSE 推送通过集成测试。
- 批量任务并发、失败隔离和重试策略通过测试。
- 结果资源可在画布、资源库和剪辑入口复用。

### Milestone D：生产工作台

范围：AI 助手、3D 导演台、剪辑预览、成片审阅。

目标：

- AI 助手可以生成 Action Plan，并在用户确认后操作画布。
- 3D 导演台可以加载基础模型、调整相机、输出参考图。
- 剪辑预览可以组织素材进入时间线。
- 成片审阅具备列表、播放、状态流转和反馈。

退出标准：

- AI 助手安全确认链路通过测试。
- Three.js 场景非空、相机和 TransformControls 可用。
- 剪辑时间线保存恢复和导出任务通过 E2E。
- 成片审阅状态机通过集成测试。

### Milestone E：商业化上线

范围：团队协作、权限、计费、后台、稳定性、安全。

目标：

- 团队、角色、权限、额度和消费记录完整。
- 后台可配置模型、查看任务、处理异常。
- 生产环境具备监控、日志、审计、安全门禁。

退出标准：

- 权限矩阵测试通过。
- 计费预检查、冻结、扣除、返还测试通过。
- 关键 API 具备限流、鉴权和审计日志。
- 安全扫描、依赖审计、E2E、性能测试通过。

## 5. Phase 级执行计划

### Phase 0：工程与基础设施

开发任务：

- 初始化 monorepo 或单仓多包结构。
- 建立前端应用、后端 API、worker、shared schema 包。
- 配置 TypeScript、ESLint、Prettier、Tailwind、测试框架。
- 配置 PostgreSQL、Redis、对象存储本地替身。
- 建立 CI：lint、typecheck、unit、build、e2e smoke。
- 建立环境变量模板和启动校验。

测试：

- `pnpm lint`
- `pnpm typecheck`
- `pnpm test`
- `pnpm build`
- 基础 Playwright smoke：应用打开、健康检查通过。

验收：

- 新开发者可以按 README 启动本地环境。
- CI 至少覆盖 lint、类型、测试、构建。
- 不存在硬编码密钥。

### Phase 1：平台基础

开发任务：

- 用户、团队、项目、画布、资源表。
- 项目 CRUD、画布 CRUD、上传资源。
- 工作空间 UI：个人项目、团队项目、最近项目、搜索。
- 资源上传：图片、视频、音频、文档。
- 基础权限：owner/admin/editor/viewer。

测试：

- Repository/API 单元和集成测试。
- 上传大小、类型、失败重试测试。
- 权限拒绝测试。
- E2E：创建项目 -> 上传资源 -> 创建画布。

验收：

- 用户可创建项目和画布。
- 项目和画布刷新后仍可恢复。
- 非授权用户无法访问项目。

### Phase 2：画布内核

开发任务：

- 接入 React Flow。
- 节点、边、viewport、selection、undo/redo 基础结构。
- 画布保存、自动保存、版本历史。
- 右键菜单、节点搜索、框选、多选、复制粘贴。
- 节点内部按钮事件隔离：`nodrag`、`nopan`、portal 下拉。

测试：

- reducer/store 单元测试。
- 画布序列化/反序列化测试。
- React Flow 组件测试。
- E2E：拖入节点、连线、保存、刷新恢复。
- 性能 smoke：100 个节点基本交互不卡死。

验收：

- 画布可以稳定保存恢复。
- 节点按钮不会触发拖拽冲突。
- 连线数据类型校验可用。

### Phase 3：节点系统骨架

开发任务：

- 建立 Node Registry。
- 覆盖 PRD/WBS 对齐表中的 31 个 `nodeId`。
- 每个节点具备默认数据、输入端口、输出端口、属性面板 schema。
- 节点执行入口统一走任务系统抽象。
- 空壳节点先不接真实模型，但必须可创建、配置、保存。

测试：

- 每个 `nodeId` 默认数据 snapshot 测试。
- 端口类型兼容测试。
- 属性面板 schema 测试。
- E2E：节点库搜索、分类、拖拽创建、配置保存。

验收：

- 31 个节点全部可创建。
- 新增节点必须先注册 `nodeId`、schema、测试和 PRD/WBS 映射。
- 节点配置变更不会破坏旧画布保存结构。

### Phase 4：剧本、主体与分镜工作流

开发任务：

- 剧本节点：粘贴、上传、编辑、版本。
- 剧本分段/优化节点：分段、情节优化、时长控制。
- 主体提取节点：角色、场景、道具结构化输出。
- 主体生成节点：主体提示词、设定图、入库。
- 分镜生成节点：分镜脚本、宫格图提示词、批量出图入口。
- 主体库：去重、别名、类型、版本、跨节点引用。

测试：

- Prompt Builder 单元测试。
- LLM JSON Schema 校验和修复重试测试。
- 主体去重和别名测试。
- E2E：剧本 -> 主体提取 -> 主体库 -> 分镜。

验收：

- 可用 mock provider 完成剧本到分镜数据闭环。
- 不清晰的节点选项已完成 Chrome/CDP 复核或列为阻断项。
- 生成结果能被后续图像节点引用。

### Phase 5：AI 模型网关与生成任务

开发任务：

- Model Gateway 和 Provider Adapter。
- 文本、图像、视频、音频任务 API。
- BullMQ 队列、Worker、状态推送。
- 任务取消、重试、失败原因归类。
- 积分预检查、冻结、扣除、失败返还。
- 任务结果写入资源库和节点历史。

测试：

- provider adapter contract test。
- 任务状态机单元测试。
- 队列集成测试。
- 积分边界测试。
- E2E：图像生成、视频生成、失败重试。

验收：

- 模型供应商可替换。
- 前端不直接理解供应商私有参数。
- 每个任务都有可追踪日志、成本和失败原因。

### Phase 6：风格与一致性系统

开发任务：

- 全局风格节点。
- 风格提示词查看、复制、编辑。
- 主体引用、资源引用、局部覆盖。
- 角色/场景/道具一致性校验。
- 从参考图提取风格。

测试：

- 风格注入规则测试。
- 主体引用解析测试。
- 不同节点局部覆盖优先级测试。
- E2E：主体 + 风格 -> 图像生成。

验收：

- 同一个主体能在多个分镜中复用。
- 全局风格能稳定注入图像/视频任务。

### Phase 7：批量生产系统

开发任务：

- 批量图像生成。
- 批量视频生成。
- 暂停、停止、重试、失败隔离。
- 队列进度、成本预估、并发控制。
- 批量下载和结果映射回节点。

测试：

- 批量任务拆分和聚合测试。
- 并发限制测试。
- 失败隔离和重试测试。
- E2E：分镜批量出图。

验收：

- 一个失败任务不会阻断整个批次。
- 批量结果和原输入顺序可追踪。
- 成本预估和实际消耗可对账。

### Phase 8：AI 创作助手

开发任务：

- 聊天面板、附件、上下文读取。
- 读取当前画布和选中节点上下文。
- Action Plan：创建节点、修改节点、连线、整理画布。
- 用户确认后执行。
- 助手操作可撤销。

测试：

- action schema 校验测试。
- 权限和安全测试。
- 确认前不执行副作用测试。
- E2E：助手生成节点操作卡片 -> 用户确认 -> 画布变更。

验收：

- 助手不能绕过用户确认修改画布。
- 所有 action 可追踪、可撤销、可审计。

### Phase 9：3D 导演台与 3D 世界

开发任务：

- Three.js 视口、相机、安全框、网格。
- GLB/GLTF/FBX/OBJ 导入。
- TransformControls：移动、旋转、缩放。
- 角色/场景/道具资源库。
- 全景背景和参考图输出。
- 3D 节点与图像/视频节点打通。

测试：

- 3D 数据序列化测试。
- Loader fallback 测试。
- Playwright canvas 非空截图测试。
- 视觉 smoke：相机、TransformControls、导出参考图。

验收：

- 3D 场景可保存恢复。
- 输出参考图可进入图像生成链路。
- 移动端或小屏至少有可用降级策略。

### Phase 10：剪辑预览

开发任务：

- 多轨时间线。
- 视频、音频、字幕、文本素材轨。
- 拖拽素材到轨道。
- 裁剪、移动、缩放、音量、预览。
- 配音、旁白、BGM。
- FFmpeg 导出任务。

测试：

- 时间线数据结构测试。
- 拖拽和裁剪组件测试。
- 导出任务集成测试。
- E2E：视频素材 -> 时间线 -> 导出。

验收：

- 剪辑数据可保存恢复。
- 导出任务可追踪状态。
- 导出结果进入成片审阅。

### Phase 11：成片审阅

开发任务：

- 剧集/成片列表。
- 播放审阅页。
- 待审、通过、驳回状态流转。
- 审阅反馈。
- 只读审阅分享链接。

测试：

- 审阅状态机测试。
- 分享权限测试。
- E2E：导出成片 -> 审阅 -> 驳回 -> 回画布修改。

验收：

- 审阅链接不暴露编辑权限。
- 审阅状态可追踪且可回溯。

### Phase 12：协作、计费与后台

开发任务：

- 团队成员、角色、项目权限。
- 协作锁、在线成员、分享链接。
- 积分包、订阅、额度、消费明细。
- 模型后台配置。
- 任务后台、审计日志、用户管理。

测试：

- 权限矩阵测试。
- 计费扣费和返还测试。
- 后台操作审计测试。
- E2E：团队成员协作、额度消耗、后台查看。

验收：

- 团队权限不会越权。
- 消费记录可对账。
- 后台修改模型配置不会破坏运行中任务。

### Phase 13：稳定性、安全与上线

开发任务：

- 监控、日志、告警。
- API 限流和安全头。
- 依赖审计、密钥扫描。
- 性能优化：画布、资源、任务队列。
- 灰度发布、回滚脚本、备份恢复。

测试：

- 安全扫描。
- 依赖审计。
- 压测和队列吞吐测试。
- 端到端回归测试。
- 灰度和回滚演练。

验收：

- 无 CRITICAL/HIGH 安全问题。
- 关键链路 E2E 全部通过。
- 生产环境具备监控、日志和回滚能力。

## 7. 第一轮落地建议

当前仓库只有 README 和文档，建议下一步不要直接实现全部业务，先完成 Phase 0 到 Phase 2 的工程骨架：

1. 建立项目结构。
2. 选择单仓多包结构：`apps/web`、`apps/api`、`packages/shared`、`packages/db`。
3. 前端启动 React/Vite/TypeScript/Tailwind。
4. 后端启动 Fastify/NestJS + health API。
5. 建立 Prisma/Drizzle schema 初版。
6. 建立测试和 CI。
7. 实现最小工作空间、项目、画布、React Flow 节点保存恢复。
8. 用 mock task provider 预留任务状态机。

第一轮验收目标：

- 本地一条命令启动前后端。
- 用户能进入工作空间并创建项目。
- 用户能进入画布，拖入 3 到 5 个代表节点。
- 节点可连线、保存、刷新恢复。
- 自动化测试覆盖核心数据结构和画布保存。

## 8. 当前阻断和待确认

- 真实登录/账号体系采用自建还是第三方 Auth。
- 第一批 AI provider 选择。
- 对象存储供应商选择。
- 是否第一版就接真实计费。
- 是否需要立刻实现多人实时协作，还是先用协作锁。
- 3D 导演台第一版支持的模型格式范围。

这些项不阻断 Phase 0 到 Phase 2，但会影响 Phase 5、9、12 的具体实现。
